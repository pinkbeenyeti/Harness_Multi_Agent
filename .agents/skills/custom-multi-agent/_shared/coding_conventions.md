# 프로젝트 코딩 컨벤션 (Coding Conventions)

> 이 문서는 모든 워커가 코드 작성 시 준수해야 하는 품질 기준입니다.
> 비평 워커도 이 기준으로 코드를 검증합니다.

## 1. 인터페이스 구현 원칙
* 한 클래스는 하나의 Port/인터페이스만 구현한다 (ISP: 인터페이스 분리 원칙).
* 2개 이상의 Port를 하나의 Adapter에 합체하지 않는다.
* 각 Port/Adapter는 단일 책임을 가진다.

## 2. 초기화 패턴
* 필드 초기화는 생성자에서 1회 수행한다.
* 매 호출마다 InitXxx() 같은 lazy 초기화 패턴을 사용하지 않는다.
* null/유효성 체크는 IsValid 같은 프로퍼티로 통합하여 메서드마다 반복하지 않는다.

## 3. 중복 코드 금지
* 2개 이상의 메서드에서 동일한 로직이 반복되면 반드시 헬퍼 메서드로 추출한다.
* Guard clause(방어 조건문)는 메서드 최상단에 1회만 배치한다.
* 동일한 EntityQuery/컴포넌트 접근 패턴이 반복되면 공통 메서드로 팩토링한다.

## 4. 코드 단순성
* 과도한 람다/스트림 중첩, 불필요한 제네릭/리플렉션을 배제한다.
* 누구나 직관적으로 읽을 수 있는 표준적이고 명확한(Idiomatic) 형태를 지향한다.
* 단, 성능 핫스팟(핵심 연산 루프, GC 민감 영역)에서는 성능을 우선한다.

## 5. 네이밍
* 클래스명은 역할을 명확히 반영한다 (예: PuzzleGamePresentationBridge).
* 메서드명은 동사로 시작하며, 반환값이 bool이면 Is/Has/Can/Try 접두사를 사용한다.
* 약어는 프로젝트 내에서 통일된 것만 사용한다.
