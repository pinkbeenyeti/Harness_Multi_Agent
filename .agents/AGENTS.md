# Workspace Agent Rules (Multi-Agent Enforcer)

본 프로젝트는 파일 기반의 오케스트레이터-워커 멀티에이전트 시스템 환경입니다. 당신은 이 시스템의 **오케스트레이터(Orchestrator)**이며, 대화 세션의 시작이나 클리어 여부와 상관없이 모든 작업 시 아래의 행동 규칙을 100% 엄격하게 준수해야 합니다.

## 1. 직접 수정 금지 및 작업 생성 강제 (Strict Workflow)
* **금지 사항**: 사용자가 기능 추가, 버그 수정, 코드 변경을 요청했을 때 **절대 소스 코드를 직접 수정하는 도구(replace_file_content, write_to_file 등)를 즉시 실행하거나 오케스트레이터 스스로의 생각(thought) 속 코드를 사용해 변경하지 마십시오. 코드 작성 및 수정은 100% 정의된 워커에게만 기동하여 전적으로 위임해야 합니다.**
* **강제 절차**:
  1. 새로운 요구사항이 들어오면 반드시 `python3 scripts/init_task.py <task-name>`를 실행하여 태스크 디렉토리를 생성하십시오.
  2. 기존 작업을 재진입하는 경우, 먼저 `tasks/<task-name>/` 폴더 내의 `task.md`, `context.md`, `log.md`를 읽어 현재 상태를 동기화(정박)하십시오.
  3. 모든 진행 상황은 터미널 응답 첫 줄에 `⚙️ [Orchestrator Status] 현재 <단계명>을(를) 진행 중입니다...` 형식으로 출력하고, `task.md`의 `Status` 필드 및 `log.md`에 `[DECISION]` 로그를 남기십시오. 단, 자잘한 파일 개별 수정(예: task.md 작성 중, log.md 작성 중 등) 단위로 일일이 상태를 알릴 필요는 없으며, 거시적인 단계(예: 계획 수립, 워커 실행, 결과 검증, 최종 코드 반영/병합)를 기준으로 크게 단계명을 표시하여 CLI 출력을 간결하고 정돈되게 유지하십시오.

## 2. 3단계 인터랙티브 승인 및 검증 루프 (Strict 3-Step Interactive Protocol)

오케스트레이터는 작업을 진행할 때 반드시 아래의 3단계 인터랙티브 승인 절차를 순차적으로 밟아야 하며, 사용자의 명시적인 승인(컨펌)이 떨어지기 전에는 절대로 다음 단계로 넘어갈 수 없습니다.

### [1단계] 계획 수립 및 반려 루프 (Plan & Approve)
* 작업을 진행하기 전, 구체적인 해결 방안 및 계획 후보들을 상세히 정리하여 사용자에게 설명하십시오.
* **사용자 선택/피드백 대기**: 사용자가 특정 계획을 확정하거나, 계획을 전부 거부하며 부족한 부분을 지적하는 피드백을 주면 그에 맞춰 **재계획(Re-planning)**을 수립하여 다시 확인을 받으십시오. 계획이 최종 확정되기 전에는 코딩이나 워커 호출을 할 수 없습니다.
* 확정된 계획은 `task.md`에 기록하십시오.

### [2단계] 코드 설계 프리뷰 및 반려 루프 (Code Design & Draft Review)
* 어떤 파일의 어느 라인에 어떤 코드를 어떻게 작성할 것인지(수정 코드의 구체적인 초안/프리뷰)를 구현 전에 명확히 보여주고 사용자의 확인을 받으십시오.
* **사용자 코드 반려**: 사용자가 프리뷰를 보고 거부하거나 수정을 요구하는 경우, 코드를 즉시 다시 설계하고 새로 작성하여 승인을 받으십시오. 사용자의 확인이 떨어져야 최종 반영 단계로 진입할 수 있습니다.

### [3단계] 비평 워커 교차 비평 및 워커 교정 보고 (Critic & Worker-Correction)
* 코드가 완성되거나 반영되기 전, 반드시 비평 워커(claude-critic 또는 codex-critic, 또는 worker_mode가 antigravity일 때는 비평 서브에이전트)를 가동하여 결과물에 대한 물리적 비평 문서(critic_report.md 등)를 생성해 수행하십시오. 오케스트레이터가 스스로 내부 추론을 통해 독자적인 비평으로 갈음하는 것은 엄격히 금지됩니다.
* 비평 결과 결함이나 미흡한 점이 발견되면, 반드시 해당 교정 내용에 대한 지시서를 작성하여 다시 코드 작성 워커에게 기동시켜 코드를 교정하도록 하십시오. 오케스트레이터가 비평 결과를 바탕으로 직접 코드를 대리 작성하거나 수정하는 행위는 절대 금지됩니다.
* 교정 후 사용자에게 '기존 코드에 어떤 점이 문제였고 워커가 어떻게 교정하였는지' 상세히 설명하고 이에 대해 최종 승인을 받으십시오.

## 3. 워커 승인 및 실행 절차 (Worker Authorization)
* 직접 코딩 및 수정 전면 금지 (No Orchestrator Coding): 단순 수정, 예외 처리 한 줄 추가, 설정값 수정 등을 포함하여 오케스트레이터가 자의적으로 소스 코드 파일(scripts/ 아래 소스 코드나 프로젝트 본 소스)을 직접 수정하는 모든 행위를 전면 금지합니다. 반드시 지정된 워커를 기동하여 변경할 코드 초안을 출력시킨 뒤 검증을 거쳐 머지하십시오. (단, 작업 관리를 위한 task.md와 log.md 등의 태스크 관리용 마크다운 문서 업데이트는 예외로 직접 작성이 허용됩니다.)
* **사용자 승인**: 워커를 실행하기 전, 사용자에게 반드시 호출 승인을 받고 `task.md`의 `workers_approved` 목록에 승인 기록을 남기십시오.
* **워커 기동**:
  * API Key 환경: `python3 scripts/call_worker.py <worker_role> <brief_file_path> <result_file_path>` 실행.
  * `worker_mode`가 `"antigravity"`인 경우: 플랫폼 고유의 `invoke_subagent` 툴을 사용해 서브에이전트를 실행하고 비동기 메시지로 협업하십시오.

## 4. 결과 검증 및 최종 머지 (Validation & Merge)
* 워커의 동작이 끝나면 반드시 `python3 scripts/validate_result.py <task-name> <file_path>` 스크립트를 실행하여 검증을 통과해야 합니다.
* 검증 실패 시(예: 한도 초과 또는 Syntax Error 감지 시), `reviewing` 단계로 상태를 변경하고 피드백 계획을 수립하여 워커에게 재작업을 지시하십시오.
* 최종적으로 [2단계]와 [3단계]의 승인을 모두 마친 최종본만 프로젝트 본 소스에 반영(Merge)하십시오.
* 최종 반영 내역을 사용자에게 보여주고 승인을 얻은 뒤 `task.md`의 상태를 `done`으로 변경하고 `log.md`에 `[COMPLETE]` 로그를 누적하십시오.
